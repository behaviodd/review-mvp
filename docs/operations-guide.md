# 운영 가이드 — 시트 보호 / 변조 방지

> **작성**: 2026-05-19 (라운드 14, B 그룹 audit 잔여 B8)
> **범위**: Google Sheets 직접 편집·변조 방지를 위한 운영자 측 보호 정책
> **연계**: `docs/stability-audit-B.md` § B8 (인사스냅샷 immutability)

---

## 1. 배경 — 코드 측 invariant 의 한계

review-mvp 의 데이터 무결성은 다음 두 축으로 보장됨:

1. **Apps Script LockService + idempotent 쓰기** — 클라이언트 경로의 race 차단 (B-1.1, B-1.2, B-2.\*)
2. **시트 직접 편집 권한 — 운영 정책으로만 보장** ← 본 문서

특히 다음 시트는 **append-only / immutable 의도**라 클라이언트에서 변조 액션 자체가 없음:

| 시트 | 의도 | 변조 시 영향 |
|---|---|---|
| `_인사스냅샷` | 사이클 발행 시점의 조직 동결 (snapshot) | 발행 후 변조 시 시점 일관성 깨짐 — 평가 결과 신뢰성 손상 (B8) |
| `_감사로그` | append-only — 모든 운영 액션 기록 | 삭제·수정 시 audit trail 깨짐 (E 그룹) |

이런 시트는 **시트 권한 + 보호 범위** 로 직접 편집을 차단해야 함.

## 2. 권장 시트 권한 설정

### 2.1 시트 단위 권한 (운영자 권한 분리)

Google Sheets 의 "공유" 메뉴에서:

| 시트 (스프레드시트 단위) | admin (앱) | 운영자 (시트 직접) | 일반 멤버 |
|---|---|---|---|
| review-mvp 마스터 SS | 편집자 | 편집자 (긴급 패치만) | 접근 차단 |

### 2.2 시트 안의 범위 보호 (Range Protection)

다음 시트는 "데이터 → 시트와 범위 보호" 로 보호 권한 설정:

#### `_인사스냅샷` 시트 (B8 핵심)
- **보호 대상**: A:Z 전체 (1행 헤더 제외)
- **편집 권한**: 본인 (Apps Script 실행 계정) 만. 운영자도 직접 편집 금지
- **변경 시 알림**: 워크북 소유자에게 이메일 자동 알림
- **이유**: 사이클 발행 시점의 시점 일관성 유지. payloadJSON 변조 시 평가 결과 무효화 가능

#### `_감사로그` 시트
- **보호 대상**: A:H 전체
- **편집 권한**: 본인 만. **삭제 차단** 우선
- **이유**: 모든 운영 액션의 audit trail. 변조 시 사용자 행동 추적 불가

#### `리뷰사이클` / `리뷰템플릿` / `리뷰제출` 시트
- **보호 대상**: 헤더 행 (1행) 만
- **편집 권한**: 본인 만
- **데이터 행**: 운영자 편집 가능 (긴급 패치 필요)
- **이유**: 헤더가 바뀌면 클라이언트 parser/writer 비대칭으로 정합성 깨짐 (라운드 13 P0-1 의 부채 패턴)

### 2.3 변조 감지

Google Sheets 의 **알림 규칙** (도구 → 알림 규칙):
- "변경사항이 있을 때마다" → 워크북 소유자 이메일
- 비정상 변경 (예: `_인사스냅샷` 의 payloadJSON 수정) 즉시 감지

## 3. 변조 발생 시 복구

### 3.1 `_인사스냅샷` 변조
1. Apps Script 의 `Migrate_R7.gs` 또는 `createSnapshot` 액션은 **append-only** — 기존 row 수정 안 됨
2. 변조 row 를 시트 history (파일 → 버전 기록) 에서 복원
3. 영향받은 사이클의 evaluator/reviewee 명단 재검증 필요

### 3.2 `_감사로그` 변조
1. 시트 history 로 복원
2. 변조 시점·범위 본 문서 § 4 의 사고 기록란에 추가
3. E 그룹 audit (관측성) phase 에서 외부 sink (Cloud Logging 등) 도입 검토

### 3.3 헤더 변조
1. Apps Script 의 `ensureHeaders(sheet, EXPECTED_HEADERS)` 가 자동으로 누락 헤더 추가 — 다만 컬럼 순서 변경은 복구 안 됨
2. 시트 history 로 복원 + 모든 row 의 컬럼 위치 재확인
3. `parseSheet*` 함수가 헤더 이름 기반 매핑이라 순서 무관하지만, raw 시각 검토 안내

## 5. 향후 개선 (별도 phase)

- **변조 감지 강화**: ETag 비교 + 정기 hash 검증 cron
- **외부 audit sink**: `_감사로그` 시트와 별개로 Cloud Logging / BigQuery 로 미러
