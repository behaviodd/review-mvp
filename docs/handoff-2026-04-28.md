## 2. 오늘 완료된 작업 (시간 순)

### 2-1. 동기화 안정성 정비 (Phase A → E)

**Phase A — `resilientFetch` 유틸 신설** (`src/utils/resilientFetch.ts`)
- timeout (20s, Vercel Edge 25s 보다 짧게) + 5xx/timeout/네트워크 오류 자동 재시도 (2회, jitter 백오프 [500, 1500]ms)
- 4xx 즉시 throw (permanent), 외부 abort 즉시 throw
- `SyncError` 클래스에 `kind: 'transient' | 'permanent' | 'timeout'` + `attempts` 보존
- 단위 13 + 실제 HTTP 서버 통합 9 = **22 테스트 (`src/utils/resilientFetch.{test,integration.test}.ts`)**

**Phase B — 동기화 훅 정비** (`useOrgSync`, `useReviewSync`)
- 모든 `fetch` → `resilientFetch` 교체
- 적응형 polling 백오프: 연속 실패 1·1·2·4·8x (cap), `setInterval` → 재귀 `setTimeout`
- `document.hidden` 시 fetch 스킵 + 다음 timer 만 예약
- visibility 복귀 시 진행 중 timer 취소 + 즉시 재시도
- **in-flight 가드** — visibility flicker 시 tick 중첩 race 차단 (트레이스 검토 중 발견·수정)
- 복구 토스트 일관 적용

**Phase E — 배너 UX 재작업** (`SyncStatusBanner.tsx` + store 확장)
- `SyncErrorKind` (`transient` / `permanent` / `timeout`) 필드 store 추가
- 배너 색상 분기: permanent = 빨강("설정/권한 확인 필요"), 그 외 = 주황("동기화 일시 지연 — 자동 재시도 중")
- 마지막 성공 시각 표시 (`{timeAgo} 마지막 동기화 성공`)
- "재시도" 버튼이 쓰기 큐 + 읽기 강제 refetch **동시** 발화 (이전엔 쓰기만)
- `src/utils/syncControl.ts` 신규 — 비-훅 위치(배너 등)에서 refetch 트리거 가능하도록 registry 패턴

**Phase D — Vercel Edge 프록시 정비** (`api/org-sync.ts`, `api/review-sync.ts`)
- GET: 18s 자체 timeout + 5xx/timeout 1회 자동 재시도 (Apps Script cold start 흡수)
- POST: 18s timeout, **재시도 안 함** (멱등성 보호 — audit.append 등 중복 위험)
- 응답 헤더 `X-Proxy-Retries` 추가 (cold start 빈도 운영 모니터링용)
- timeout 시 502 → 504 정확히 분류

**Phase C-extras — 멱등 쓰기 retry** (`sheetWriter.ts`, `reviewSheetWriter.ts`)
- `IDEMPOTENT_ACTIONS` 세트로 액션별 retry 정책 결정
  - upsert/delete/end 류 → retries=1
  - createUser(자동 ID)/createSnapshot/logImpersonationStart/appendAudit → retries=0
- 쓰기 timeout 25s (프록시 18s 보다 충분히 길게)

## 10. 알려진 함정·주의사항

1. **`useReviewSync`/`useOrgSync` 는 App.tsx 의 OrgSyncProvider/ReviewSyncProvider 에서만 mount** — 다른 곳에서 다시 호출하면 polling 중복. Settings 같은 페이지는 `refetchOrg`/`refetchReview` (syncControl) 만 사용.
2. **resilientFetch 의 4xx 는 permanent throw** — 호출자는 `if (!res.ok) throw` 가 필요 없음. 단 Apps Script 의 body-level error (200 응답 + `{error: ...}`) 는 별도 throw 필요.
3. **markPendingWrite 는 두 번 호출이 정상** (fetch 전·후, sheetWriter.ts 주석 참조).
4. **POST 는 멱등 액션만 retry** — `IDEMPOTENT_ACTIONS` 세트 외 액션 (createUser, appendAudit 등) 은 retries=0.
5. **인사스냅샷 시트** 가 시점 의존 조직도의 단일 진실 — 검증 시 `snapshotResolver.ts` 함께 봐야.
6. **Vercel Edge functions 는 별도 배포** — `npm run build` 는 검증하지 않음. `api/*.ts` 변경은 push 후 Vercel 배포로만 검증 가능.
7. **bulkGetAll 미배포 폴백** — Apps Script 가 `bulkGetAll` 액션 미지원이면 6개 병렬 fetch 로 자동 폴백 (`useOrgSync.ts` 의 `fallbackModeRef`). 폴백 모드 진입 시 콘솔에 1회 경고.
8. **`LockService`** — Apps Script 측 동시성 가드. B 그룹 audit 시 모든 doPost 분기에서 적절히 사용되는지 확인 대상.
